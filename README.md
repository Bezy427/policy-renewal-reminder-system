# 🛡️ Policy Renewal System — Insurance Policy Renewal Reminder System

A lightweight internal web application for insurance company staff to manage policies, track renewal dates, upload supporting documents, and view portfolio insights through a live dashboard.

## Setup Instructions

### Requirements

Before you begin, make sure your environment has:

| Requirement | Minimum Version |
|-------------|----------------|
| PHP | 8.0+ |
| MySQL or MariaDB | 5.7+ / 10.4+ |
| Apache | With `mod_rewrite` enabled |
| PHP Extensions | `pdo_mysql`, `fileinfo`, `mbstring` |

> **XAMPP users:** All of the above are included. Just make sure Apache and MySQL are running in the XAMPP Control Panel.

---

### Step 1 — Copy files to your server

**XAMPP (Windows):**
```
C:\xampp\htdocs\policy-system\
```

**XAMPP (Mac):**
```
/Applications/XAMPP/htdocs/policy-system/
```

**Linux / Apache:**
```
/var/www/html/policy-system/
```

---

### Step 2 — Create the database

Open **phpMyAdmin** (`http://localhost/phpmyadmin`) and run the contents of:

```
config/schema.sql
```

This creates the `policy_renewal_db` database, all tables, and sample policy records.

Alternatively, via the command line:

```bash
mysql -u root -p < config/schema.sql
```

---

### Step 3 — Configure database credentials

Open `config/config.php` and update the following lines to match your environment:

```php
define('DB_HOST', 'localhost');
define('DB_NAME', 'policy_renewal_db');
define('DB_USER', 'root');       // your MySQL username
define('DB_PASS', '');           // your MySQL password
```

> `APP_URL` is auto-detected from the server request — you do not need to set it manually.

---

### Step 4 — Set upload folder permissions

The `uploads/` folder must be writable by the web server.

**Linux / Mac:**
```bash
chmod 755 uploads/
```

**XAMPP on Windows:** This is handled automatically — no action needed.

---

### Step 5 — Seed default users

The schema creates the database tables but password hashes must be generated by your own PHP installation to guarantee compatibility.

1. Open your browser and visit:
   ```
   http://localhost/policy-system/setup.php
   ```
2. Click **"Seed Users & Set Passwords"**
3. Wait for the success confirmation
4. Click the link to go to the login page

> ⚠️ **Delete `setup.php` from your server after this step.** It is a security risk if left in place.

---

### Step 6 — Log in

| Role | Email | Password |
|------|-------|----------|
| Administrator | admin@company.com | Admin@1234 |
| Policy Officer | officer@company.com | Officer@1234 |
| Viewer | viewer@company.com | Viewer@1234 |

> Change all default passwords after your first login.

---

### Troubleshooting

| Problem | Fix |
|---------|-----|
| Blank page or HTTP 500 | Visit `diag.php` in your browser for a full diagnostic report |
| Cannot log in | Run `setup.php` to regenerate password hashes on your server |
| Uploads not saving | Check that the `uploads/` folder exists and is writable |
| Pages not found | Ensure Apache `mod_rewrite` is enabled and `AllowOverride All` is set |

> Delete `diag.php` from the server once your setup is confirmed working.

## Project Structure

The application follows a **manual MVC (Model-View-Controller)** pattern with a single front controller. Every request enters through `index.php`, which reads the URL parameters and routes to the appropriate controller.

```
policy-system/
│
├── index.php                     # Front controller — all requests route here
├── setup.php                     # First-run user seeder (delete after use)
├── diag.php                      # Environment diagnostic tool (delete after use)
│
├── config/
│   ├── config.php                # Global constants: DB credentials, upload limits, app settings
│   ├── Database.php              # PDO singleton — one shared database connection
│   ├── schema.sql                # Full database schema with seed data
│   └── reset_passwords.sql       # Standalone SQL to reset user passwords if locked out
│
├── models/                       # Data entities — represent a single database row
│   ├── User.php                  # User entity with role helper methods
│   ├── Policy.php                # Policy entity with renewal calculations and status labels
│   └── Document.php              # Document entity with file size formatting and icon helpers
│
├── services/                     # Business logic — all database queries and processing live here
│   ├── AuthService.php           # Login, logout, session lifecycle, CSRF, role enforcement
│   ├── PolicyService.php         # Policy CRUD, filtering, dashboard summary counts, validation
│   ├── DocumentService.php       # File upload validation, storage, download, deletion
│   ├── UserService.php           # User CRUD and validation (admin only)
│   └── ActivityLogger.php        # Writes audit entries to the activity_log table
│
├── controllers/                  # HTTP layer — receives requests, calls services, loads views
│   ├── PolicyController.php      # Routes: list, view, create, edit, delete
│   ├── DocumentController.php    # Routes: upload, download, delete
│   └── UserController.php        # Routes: list, create, edit, toggle active
│
├── views/                        # HTML templates — display data only, minimal logic
│   ├── partials/
│   │   ├── header.php            # Shared HTML head, sidebar, topbar
│   │   └── footer.php            # Closing tags and JavaScript includes
│   ├── auth/
│   │   └── login.php             # Login form
│   ├── dashboard/
│   │   └── index.php             # Summary stats and nearing-renewal table
│   ├── policies/
│   │   ├── list.php              # Filterable, searchable policy table
│   │   ├── form.php              # Shared create/edit form
│   │   └── view.php              # Policy detail with document management
│   └── users/
│       └── index.php             # User table with create/edit modals
│
├── assets/
│   ├── css/style.css             # Full application stylesheet (dark theme, CSS variables)
│   └── js/app.js                 # UI interactions: sidebar, drag-drop upload, search, modals
│
└── uploads/                      # Stored document files — direct browser access blocked
    └── .htaccess                 # Denies all direct requests to this folder
```

### How a Request Flows

```
Browser request
      ↓
  index.php          ← reads ?page= and ?action= from URL
      ↓
  Controller         ← checks authentication and role, calls service(s)
      ↓
  Service            ← runs validated, prepared SQL queries, returns data
      ↓
  Model              ← wraps database rows into typed objects
      ↓
  View               ← renders HTML using the model data
      ↓
Browser response
```

## Role Design

The system has three roles. Permissions are enforced **server-side in the controllers** — not just hidden in the UI.

### Roles

**Administrator**
- Full access to everything in the system
- The only role that can create, edit, and deactivate user accounts
- Can see all policies and documents regardless of who created them

**Policy Officer**
- The primary working role for day-to-day operations
- Can add, edit, and delete policies
- Can upload and delete documents attached to policies
- Cannot access user management

**Viewer**
- Read-only access
- Can browse the dashboard, policy list, and policy details
- Can download documents but cannot upload or delete them
- Cannot modify any data in the system

### Permissions Matrix

| Feature | Admin | Policy Officer | Viewer |
|---------|:-----:|:--------------:|:------:|
| View dashboard | ✅ | ✅ | ✅ |
| View policy list | ✅ | ✅ | ✅ |
| View policy detail | ✅ | ✅ | ✅ |
| Add policy | ✅ | ✅ | ❌ |
| Edit policy | ✅ | ✅ | ❌ |
| Delete policy | ✅ | ✅ | ❌ |
| Upload documents | ✅ | ✅ | ❌ |
| Download documents | ✅ | ✅ | ✅ |
| Delete documents | ✅ | ✅ | ❌ |
| View user list | ✅ | ❌ | ❌ |
| Create users | ✅ | ❌ | ❌ |
| Edit users | ✅ | ❌ | ❌ |
| Activate / deactivate users | ✅ | ❌ | ❌ |

### How Enforcement Works

Role checks happen in the controller before any action is taken:

```php
// In PolicyController — only admins and officers can create/edit/delete
$this->authService->requireRole($currentUser, ['admin', 'policy_officer']);
```

If a user with the wrong role hits a restricted route directly via URL, they are redirected to the dashboard. The UI also hides buttons they cannot use, but this is cosmetic — the backend enforcement is what actually protects the data.

## Assumptions Made

The following decisions were made where the brief did not specify exact behaviour:

**Authentication**
- A simple email and password login was implemented. No password reset flow was built as it was not in the brief. Admins can update any user's password from the User Management screen.
- Sessions expire after **30 minutes** of inactivity as a reasonable security default for an internal office tool.

**Policy statuses**
- The three statuses (Active, Expired, Pending Renewal) are manually set by the Policy Officer — the system does not automatically change a policy's status when its renewal date passes. This was assumed intentional, as an officer may need to keep a policy marked Active while a renewal is being processed.
- Policies nearing renewal within **30 days** are flagged on the dashboard. This threshold is configurable in `config/config.php` via the `RENEWAL_NOTICE_DAYS` constant.

**Policy deletion**
- Deleting a policy also deletes all its linked documents (both database records and physical files). This is enforced at the database level via a `CASCADE` foreign key constraint.

**Document storage**
- Uploaded files are stored inside the project's `uploads/` folder with randomised filenames. In a production environment it would be preferable to store them outside the web root entirely. This was not done here to keep the setup simple for a local development environment.
- Maximum file size is set to **5 MB**. This can be changed in `config/config.php`.

**User management**
- Admins cannot delete users — only deactivate them. This preserves the audit trail and the `created_by` references on policies. A deactivated user cannot log in but their historical data remains intact.
- An admin cannot deactivate their own account to prevent being locked out.

**Search and filtering**
- The policy list supports filtering by status and searching by policy number or client name. Pagination was not implemented as it was not in the brief and the expected data volume for an internal tool is manageable without it.

**Insurance types**
- A predefined dropdown list of common insurance types was used rather than a free-text field, to ensure data consistency across the system.
